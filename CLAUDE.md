# 소통과 출력

## 여러 내용을 한꺼번에 전달하게 될 때

한 응답에 여러 항목·선택지·정보·설명을 주루룩 늘어놓게 되는 경우, 그렇게 한꺼번에 다 내놓지 말고 다음 순서로 진행한다.

1. 먼저 **요약본만** 나열한다 (예: "관련된 건 3가지다: A, B, C").
2. 그다음 **한 항목씩** 제시하며 사용자와 함께 하나씩 해결해 나간다.
3. 질문은 **한 번에 하나만** 한다.

## 질문할 때는 선택지를 주되 "그 외"를 열어 둔다

사용자에게 결정을 물을 때 막연히 열린 질문만 던지지 않는다.
구체적 선택지를 제시해 고를 기준을 준다 -- 선택지가 전혀 없으면 결정이 더 어렵다.
동시에 그 선택지에 없는 답도 받도록 "그 외(something else)"를 항상 열어 둔다.
선택지를 정해진 목록으로만 한정하면 사용자가 실제로 원하던 다른 답이 드러나지 못한다.

## 응답은 존댓말로

사용자와의 모든 대화(설명, 질문, 완료 보고)는 한국어 존댓말로 작성한다.
반말, 평어체를 쓰지 않는다.
파일 내용(README, 스킬 문서 등)의 문체는 별개이며 해당 파일 자체 규칙을 따른다.

## 콘솔 출력 형식 (ASCII 만)

이 규칙은 **대화창(콘솔)에 출력하는 응답에만** 적용한다.
파일 내용(README, 스킬 문서 등)은 대상이 아니며, 그런 파일은 각자의 규칙을 따른다
(스킬의 출력 형식은 스킬 안에서 정의한다 -- 이 규칙을 거기까지 확대하지 않는다).

대화창 응답에서는 유니코드 장식 문자를 쓰지 않고 ASCII 로만 쓴다(제목, 목록, 표 포함).
자주 빠뜨리는 항목과 치환:

- 화살표(오른쪽/왼쪽 화살표 기호)        -> `->`, `<-`
- em대시, en대시                        -> `--`
- 곡선 따옴표(스마트 쿼트)              -> 곧은 따옴표 `"` `'`
- 말줄임표(한 글자짜리 …)               -> `...`
- 가운뎃점, 장식 불릿(•, ◦ 등)          -> `-`
- 체크/엑스 표시, 이모지                 -> 쓰지 않고 말로 (OK, FAIL, `[x]`, `[ ]`)
- 박스드로잉 선문자                      -> 코드블록 안에서도 `-`, `|`, `+`
- 비교기호의 유니코드형(≤ ≥ ≠)          -> `<=`, `>=`, `!=`

"장식이 아니라 의미 기호"라는 이유로도 예외를 두지 않는다.
한글 본문(의미 텍스트)은 그대로 둔다. markdown 표의 `|` 같은 ASCII 문자는 허용한다.

# 작업 진행 방식

멀티스텝 작업은 기본적으로 사용자 확인 없이 끝까지 자율로 진행한다.
생성, 편집, 리네임, 검색, 에이전트 호출, 렌더 등 중간 단계마다 멈춰 묻지 않는다.

다음은 먼저 확인받는다.

- 기존 파일 덮어쓰기로 원래 내용이 통째로 사라지는 경우
- 여러 파일 일괄 수정 등 되돌리기 어려운 대규모 변경
- 파일/디렉토리 삭제 (절차는 아래 "삭제" 참조)

설계/구조 결정처럼 사용자 판단이 필요한 갈림길은 위와 별개로 항상 묻는다.
git 커밋/푸시와 외부 전송(PR, 메일, 메시지 등)은 사용자가 요청할 때만 한다.

## 삭제

삭제 규칙은 여기로 모은다 -- 다른 섹션(메모리 prune 등)은 이 규칙을 참조한다.

- 임시파일/스크래치: 자율로 삭제한다(확인 없음).
- 그 외 파일/디렉토리: 삭제 전 사용자 확인을 받는다.
- 사용자가 삭제를 지시하면 되묻지 말고 바로 한다(매 턴 "지울까요?" 떠보지 않는다).
- 메모리 prune 도 이 규칙을 따른다(메모리 관리 섹션 참조).

# 메모리 관리

메모리(프로젝트별 `memory/` 디렉토리)에는 point-in-time 상태만 담는다 -- 프로젝트 스코프이고
repo 로는 도출되지 않는 것. 아래 용도로만 쓴다.

- 작업 히스토리 / 진행 상태 (T-리스트, PR 상태, 완료·보류 플래그)
- 보류 + 로드맵 (열린 결정, 다음 단계)
- 결정·맥락 기록 (왜 이렇게 했는지 -- repo·git 에 안 남는 배경)
- 미착수 아이디어 / 방향
- 크로스-세션·크로스-레포 핸드오프
- 레퍼런스 (외부 URL·티켓·대시보드)
- 허브 (열린 보류 항목들을 묶는 상위 노드)

메모리에 쓰지 않는 것.

- 피드백형(= Claude 가 어떻게 일해야 하는지에 대한 규칙)은 메모리가 아니라 CLAUDE.md 에 둔다.
  보편이면 전역(Hortus), 특정 프로젝트 한정이면 그 프로젝트 CLAUDE.md. 메모리 타입
  user/feedback/project/reference 중 feedback 이 이렇게 빠지므로, 메모리는
  project/reference/user 중심이 된다.
- repo 가 이미 기록하는 것(코드 구조·과거 수정·git 히스토리·CLAUDE.md), 이 대화에서만 의미 있는 것.

## prune -- 끝난 메모리는 미루지 말고 정리

작업이 끝나 불필요해진 메모리는 미루지 말고 바로 정리 대상으로 식별한다. "완료됐지만 혹시
몰라 둔다"로 쌓아두지 않는다. 다만 실제 삭제는 "작업 진행 방식 > 삭제" 규칙과 동일하게
사용자 확인을 받고 진행한다 -- 확인 없이 자율로 지우지 않는다. 삭제할 때 다른 메모리의
`[[링크]]` 와 MEMORY.md 인덱스 줄도 함께 정리한다.

- 왜: (1) 잔여 메모리는 소량이라도 매 세션 컨텍스트를 소모한다. (2) 더 큰 비용 -- 끝난 일이
  남아 있으면 아직 할 일로 오인된다.
- 단서: "완료=무조건 삭제"가 아니라 성격을 먼저 본다. (a) 살아있는 규칙·레퍼런스, (b) 열린
  보류 항목이 참조하는 허브, (c) 다른 메모리가 실질 참조하는 것은 남긴다. 지울 때는 inbound `[[링크]]`
  를 grep 으로 먼저 찾아 함께 정리한다.

# 수정·실행 위치 규칙

아래 규칙은 모두 특정 작업을 정해진 레포/위치에서만 하도록 하는 규약이다.
현재 위치가 어디인지는 git remote 로 판정한다.

## 전역 설정 파일 (Hortus)

전역 설정 파일(전역 CLAUDE.md, settings.json 등)의 실체는 Hortus 레포
(remote `Olbbemi/Hortus`)에서 관리하며, `~/.claude` 의 해당 파일들은 그 레포 파일을
가리키는 심볼릭 링크다. 링크는 Hortus 루트의 `install.sh` 가 생성한다 -- 새 장비에서는
Hortus 를 클론하고 `install.sh` 를 한 번 실행하면 동일한 전역 설정이 구성된다.

수정은 반드시 Hortus working tree 안에서만 한다. 현재 위치가 Hortus 인지는 git remote
가 `Olbbemi/Hortus` 인지로 판단한다 -- 다른 remote 이거나, git 레포가 아니라 remote
조회 자체가 실패하면 "Hortus 아님"으로 보고 전역 설정을 그 자리에서 고치지 않는다.

- Hortus 밖에서는 전역 CLAUDE.md / settings.json 을 직접 수정하지 않는다 (Edit/Write/Bash 모두).
- 고칠 내용이 생기면 메모리에 적어두고, Hortus 루트에서 작업할 때 반영한다.
- PreToolUse 가드 훅(`hooks/guard-global-config.sh`)이 Edit/Write/NotebookEdit 의 대상
  경로를 realpath 로 정규화해 관리 대상이면 기계적으로 막는다. 다만 Bash 를 통한 수정
  (sed/tee/echo>/cp/mv 등)은 훅이 완벽히 못 잡으니, 이 규칙이 그것을 막는 최종 장치다.
- 수동 에디터로 직접 여는 것까지는 막지 못한다. 이 장치의 목적은 다른 세션에서의 무분별한
  수정 방지이며, 사용자 본인의 수동 편집은 범위 밖이다.

## armarium 플러그인 스킬

armarium 플러그인(Hortus 마켓플레이스)의 스킬을 수정해야 할 때는, 어느 프로젝트에서
작업 중이든 반드시 armarium 레포 원본 working tree 에서 수정한다.

현재 위치가 armarium 원본인지는 git remote 로 판단한다 -- remote 가 `Olbbemi/Armarium`
이면 원본이다. 설치된 캐시는 .git 이 없어 remote 조회에서 자연히 배제된다.

- armarium 원본 레포가 아니면 그 자리에서 스킬을 직접 고치지 않는다.
- 고칠 내용을 메모리에 적어두고, armarium 루트에서 작업할 때 반영한다.
- 설치된 플러그인 캐시(`~/.claude/plugins/cache/...`)는 원본의 하류 산출물이라 수정 대상이 아니다.

## knowledge-promote 실행

knowledge-promote 는 반드시 Herbarium 프로젝트 루트에서 실행한다 (wip/·knowledge/ 가 거기 있음).
루트 판정은 git remote 가 `Olbbemi/Herbarium` 인지로 한다. 아니면 먼저 그 루트로 이동한다.
